# 需求漂移防护

## 核心流程

每次触发时执行 3 步：**定位 → 比对 → 报告**。

### Step 1: 定位相关文档

读取 `references/doc-map.md`，根据当前任务所属维度，找到需要比对的 1-3 份文档。

维度速查：

| 当前任务涉及 | 必查文档 |
|-------------|---------|
| 范围判断（做不做） | `v1实现总合同.md` §5 必做 / §8 不纳入 |
| 状态字段 | `v1实现总合同.md` §4 状态合同 |
| 时间段 | `v1实现总合同.md` §3 时段合同 |
| 事件/调度 | `事件权重与调度规则.md` + `JSON数据格式规范.md` |
| 角色行为 | 对应角色档案 `docs/设定/角色/X.md` + `人设设定.md` |
| 战斗 | `v1实现总合同.md` §6 + `战斗系统设计.md` |
| UI/界面 | `界面页签与常驻信息设计.md` |
| 商店/物品 | `预算与商店刷新设计.md` |
| 地图 | `地图与地点开放设计.md` |
| 结算 | `结算与反馈系统设计.md` |

**优先级规则**：`v1实现总合同.md` > `第二章最小Demo实现清单.md` > 其他系统文档。冲突时以高优先级为准。

### Step 2: 比对 5 个检查维度

逐项检查，跳过不相关的维度：

**D1 - 范围边界**
- 此功能是否在 v1 必做清单内？
- 是否不小心引入了"暂不做"清单中的内容？
- 复杂度是否超过 Demo 所需的最小程度？

**D2 - 数据契约**
- 使用的状态字段名是否与 §4 状态合同一致？（如 `coverIntegrity` 而非自创名）
- JSON 数据结构是否符合 `JSON数据格式规范.md`？
- 事件字段是否包含必须的 `preconditions` / `effects` / `script`？

**D3 - 流程一致**
- 时间段是否严格遵循五段制？（morning / day / after_school / night / pre_sleep）
- 事件是否在正确的时段触发？
- 日结算/睡前回收的流程是否与设计文档吻合？

**D4 - 角色行为**
- 角色言行是否符合人设档案？
- 桃宫蜜 → 温暖但增加依恋代价
- 梨原澄 → 有效帮助但逼近真相
- 邪神 → 最强捷径也是最大风险
- 关系事件的冷却和阶段是否遵循规则？

**D5 - 技术一致**
- 代码结构是否与 `项目技术文档.md` 描述的架构一致？
- 新模块是否放在正确的目录？（core/ render/ ui/）
- 页面路由是否通过 `GameState.currentPage` 管理？
- 渲染是否走 NanoVG Mode B 管线？

### Step 3: 输出报告

格式：

```
## 漂移检查报告

**任务**: [当前任务简述]
**比对文档**: [实际读取的文档列表]

### 检查结果

| 维度 | 状态 | 说明 |
|------|------|------|
| D1 范围 | ✅/⚠️/❌ | ... |
| D2 数据 | ✅/⚠️/❌ | ... |
| D3 流程 | ✅/⚠️/❌ | ... |
| D4 角色 | ✅/⚠️/❌ | ... |
| D5 技术 | ✅/⚠️/❌ | ... |

### [如有问题] 漂移项

1. **[漂移描述]**
   - 文档要求: ...
   - 当前实现: ...
   - 建议修正: ...
```

状态含义：
- ✅ 一致，无问题
- ⚠️ 轻微偏差，可接受但需记录
- ❌ 明确违反文档约定，必须修正后再继续

## 自动触发规则

以下情况**必须**在写代码前执行漂移检查：

1. **新建文件** — 检查 D1（是否在范围内）+ D5（目录是否正确）
2. **新增状态字段** — 检查 D2（字段名是否合同中有）
3. **涉及时间段** — 检查 D3（是否五段制）
4. **涉及角色对话/事件** — 检查 D4（人设是否匹配）
5. **累计修改 > 200 行** — 全量检查 D1-D5
6. **用户明确要求** — 全量检查 D1-D5

## 常见漂移模式

避免以下已知的漂移陷阱：

| 漂移模式 | 典型表现 | 对照文档 |
|---------|---------|---------|
| 范围蔓延 | 顺手加了"完整失败演出"等 v1 不做的功能 | v1合同 §8 |
| 字段自创 | 用 `disguiseLevel` 代替合同中的 `coverIntegrity` | v1合同 §4 |
| 时段混乱 | 用四段制或自创时段名 | v1合同 §3 |
| 角色失焦 | 梨原澄变得过于温暖，模糊了与桃宫蜜的差异 | 角色档案 |
| 过度工程 | 为 Demo 实现了完整多结局系统 | Demo清单 §2.3 |
| 数据越界 | 事件 JSON 加了规范中不存在的字段 | JSON规范 |
